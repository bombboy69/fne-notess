1. Fog vs Cloud vs Edge (The Continuum)

This is the foundational paradigm of the module. You must be able to contrast these layers critically in terms of latency, bandwidth, and computational placement.

[ LAYER 4: CLOUD COMPUTING ] Location: Remote Data Centers (AWS, Azure) | Latency: >100ms | Compute: Infinite Function: Long-term Big Data storage, heavy ML training, global dashboards. ▲ (WAN / Fiber / Satellite) ▼ [ LAYER 3: FOG COMPUTING ] Location: Network Infrastructure (ISPs, Cell Towers, Regional Routers) Latency: 10ms - 50ms | Compute: High (Server racks) Function: Mobility support, geographical aggregation, distributed intelligence. ▲ (LAN / 5G / Ethernet) ▼ [ LAYER 2: EDGE COMPUTING ] Location: 1-Hop from Device (IoT Gateways, Cloudlets, Smart Routers) Latency: 1ms - 10ms | Compute: Moderate Function: Protocol translation (BLE to IP), local data filtering, fast actuation. ▲ (PAN / BLE / Zigbee / LoRa) ▼ [ LAYER 1: MIST COMPUTING (PERCEPTION) ] Location: On-Device (Microcontrollers, Sensors) Latency: <1ms | Compute: Extremely Low Function: Raw data sensing, basic threshold logic, ADC conversion.
Feature Cloud Computing Fog Computing Edge Computing
Mobility Support Limited (Connection drops when moving) High (Handles cellular handoffs) Low (Confined to local Wi-Fi/LAN)
Data Processing Centralized Batch Analytics Hierarchical / Distributed Analytics Local Filtering / Immediate Inference
Bandwidth Cost High (Ingress fees for raw data) Low (Aggregates and compresses data) Lowest (Filters out noise locally)

🎯 How to write this in exams

Examiner Keywords: Compute Continuum, Distributed Intelligence, Geographical Aggregation.

Common Mistake: Confusing Edge and Fog. Edge is highly localized (a gateway in a room). Fog is hierarchical and deeply integrated into the network infrastructure (ISP routers, 5G towers), making Fog far superior for Mobility Support (e.g., tracking a connected ambulance across a city).

2. IoT + Fog + Cloud Integration Pipeline

You must illustrate how data transforms as it moves from the physical world to the cloud.

THE DATA AGGREGATION & PROTOCOL WORKFLOW [SENSORS] ----(Non-IP: Zigbee/BLE)----> [EDGE GATEWAY] ----(IP: MQTT/TCP)----> [CLOUD] 1. SENSE: Sensor samples vibration at 1000Hz. 2. TRANSMIT: Sends raw analog-to-digital data via low-power Zigbee. 3. TRANSLATE: Gateway receives Zigbee, strips payload, packages into IP/TCP. 4. FILTER (Selectivity): Gateway runs Edge AI (FFT). Drops 99% of normal vibration data. 5. AGGREGATE: Creates a 1KB JSON metadata alert: {"status": "bearing_failure"}. 6. PUBLISH: Sends alert via MQTT to the Cloud. 7. ACTUATE: Cloud updates digital twin; Gateway instantly shuts down the local machine.

3. Scenario-Based System Design (The 50-Markers)

This is the most critical section. You will be asked to design an architecture for a specific scenario. Apply this exact structure.

Scenario A: Smart Agriculture (Precision Farming)

  • The Problem: 10,000 acres, battery-powered soil sensors, no Wi-Fi, spotty rural internet.
  • Fog Justification (BLURS): Cloud-only fails due to Uninterrupted Transmission (if rural internet drops, crops die) and Bandwidth (sending raw data from 10k sensors to AWS is too expensive). Fog enables local, offline actuation of irrigation.
  • Protocols: LoRaWAN (Class A) for Sensors to Gateway (Sub-GHz penetrates wet crops up to 15km; Class A maximizes 10-year battery life). MQTT from Gateway to Cloud (reliable telemetry delivery).
  • Security: AES-128 encryption over LoRaWAN. Physical tamper-evident seals on sensors. Secure Boot on the Gateway to protect AWS keys.
  • Assumptions: Assume gateway has solar+battery backup; sensors run strictly on coin-cells.

Scenario B: Connected Ambulance (Emergency Healthcare)

  • The Problem: Streaming HD ultrasound and ECGs from a moving ambulance to a hospital.
  • Fog Justification: Cloud fails due to Latency (doctors need sub-millisecond real-time video) and Mobility (Edge Wi-Fi drops when driving). Fog MEC (Multi-access Edge Compute) at 5G cell towers provides seamless handoffs and ultra-low latency.
  • Protocols: BLE for wearable ECGs (low power). Wi-Fi 6 for HD Ultrasound to onboard router. 5G URLLC for Gateway to Fog. UDP for video (TCP retransmissions would cause lag). MQTT (QoS 2) for vitals (cannot drop a heartbeat).
  • Security & GDPR: mTLS (Mutual TLS) between ambulance and hospital. Edge AI anonymizes patient faces in video before transmission (Privacy by Design).

🎯 20-Mark Answer Structure Template

  1. BLURS Justification: Prove why the Cloud fails here (Latency, Bandwidth, Offline capability).
  2. 4-Tier Architecture: List Mist (Sensors) → Edge (Gateway) → Fog (Regional) → Cloud.
  3. Protocol Elimination: "I chose LoRaWAN because Wi-Fi fails on range and 4G fails on battery."
  4. Aggregation Strategy: State exactly *what* data is dropped at the Edge.
  5. Assumptions: Always write 1-2 engineering assumptions at the bottom.

4. Protocol Selection Guide (The Iron Triangle)

You cannot have Range, Bandwidth, and Battery Life simultaneously. Protocol selection defines your system.

Network Layer: BLE vs LoRaWAN

Feature BLE 5.0 (Bluetooth) LoRaWAN
Range Short (10m - 50m) Long (Up to 15km outdoors)
Bandwidth High for IoT (~2 Mbps) Very Low (0.3 kbps - 50 kbps)
Use Case Wearable healthcare, smart home. Agriculture, smart city meters.

Application Layer: MQTT vs CoAP

Feature MQTT CoAP
Transport Base TCP (Connection-oriented) UDP (Connectionless)
Architecture Publish/Subscribe (Requires Broker) Client/Server (RESTful: GET/POST)
Battery Impact High (TCP 3-way handshake overhead) Ultra-Low (Fire-and-forget UDP)
Best For... Reliable networks, continuous power. Lossy networks (6LoWPAN), battery power.

🎯 Exam Protocol Cheat Sheet

If prompt says "Lossy network, battery-powered" → Select CoAP (UDP) to avoid TCP retransmission storms.
If prompt says "Financial/Medical data, reliable network" → Select MQTT (TCP) with QoS 2 (Exactly Once delivery).
If prompt says "City-wide or Farm, battery sensors" → Select LoRaWAN or NB-IoT.

5. Fog Advantages (Beyond Latency)

If asked for 4 advantages other than latency, use these:

  1. Bandwidth Optimization: Edge nodes filter and compress raw telemetry, sending only semantic metadata to the cloud, saving massive WAN ingress costs.
  2. Uninterrupted Operation (Resilience): Local fog nodes maintain local databases and control logic. If the internet drops, the factory robotic arms continue to operate safely offline (Edge Autonomy).
  3. Context Awareness & Mobility: Fog nodes situated in telecom cell towers know exactly where a connected car is and can seamlessly hand off the computing session to the next tower as the car drives, which localized Edge Wi-Fi cannot do.
  4. Security & Privacy Containment: Fog allows sensitive data (like patient health records) to be processed and anonymized entirely within the local hospital firewall (Data Residency) before reaching the public cloud, ensuring GDPR/HIPAA compliance.

6. The 13Vs of IoT Big Data

Move beyond Volume, Velocity, and Variety. Focus on these IoT-specific challenges:

  • Veracity: The trustworthiness of data. Sensors break or get dirty. A dirty thermometer sends false data. IoT systems must handle this uncertainty (Byzantine faults).
  • Volatility: How fast data expires. A sensor detecting a car crash is highly volatile; its value expires in 2 seconds. Cloud processing is too slow for volatile data.
  • Vulnerability: IoT devices lack CPU power for heavy encryption, making them prime targets for botnets (e.g., Mirai) and MITM attacks.
  • Validity: Data might be accurate (Veracity) but applied to the wrong model (e.g., measuring vibration on the wrong axis of a machine).

7. Security Architecture & GDPR

Hardware Security & Boot

Software security fails if the hardware is compromised. Fog gateways must use a Hardware Root of Trust (TPM) to store cryptographic keys safely. They must implement Secure Boot: verifying the cryptographic signature of the operating system before booting to ensure no malware has been installed via physical tampering.

Data States

  • Data in Motion: Secure using TLS (for TCP) or DTLS (for UDP/CoAP).
  • Data at Rest: AES-256 Full Disk Encryption on the Edge Gateway.

GDPR & Privacy

Apply Privacy by Design. Apply Data Minimization: Edge nodes should strip Personally Identifiable Information (PII) like faces or license plates *locally* before transmitting video to the cloud.

8. Middleware & Web of Things (WoT)

The Problem: IoT is heavily fragmented. A Zigbee Philips bulb cannot talk natively to an IP-based Google Cloud. Middleware (like Azure IoT Edge) acts as the protocol translator and local orchestrator.

The Web of Things (WoT): Sits at Layer 7 (Application). It doesn't invent new radio protocols; it uses standard web technologies (HTTP, JSON) to abstract hardware.

  • Thing Description (TD): The index.html of a smart device. A JSON-LD document that describes the device's Properties (Read temp), Actions (Turn on motor), and Events (Fire alarm!).
  • Semantic Interoperability: JSON-LD ensures that different systems agree on the *meaning* of data (e.g., mapping "temp" and "temperature" to the exact same W3C ontology definition).

9. Edge Native vs Cloud Native

Cloud Native Edge Native
Assumes perfect, infinite WAN connectivity. Designed for intermittent connectivity (Autonomous).
Hardware Agnostic (runs on any VM). Hardware Aware (needs specific sensors/GPUs).
Stateless (relies on cloud databases). Stateful (maintains local cache if offline).

10. 5G URLLC

URLLC (Ultra-Reliable Low-Latency Communication) is a 5G slice designed specifically for mission-critical IoT (not downloading movies). It guarantees <1ms latency and 99.999% reliability. It is mandatory for Autonomous Vehicles (V2X), Remote Telesurgery, and real-time industrial robotics.
